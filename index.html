Background

What is
Business Intelligence (BI)?

BI turns raw data into actionable insight. — provided you already know exactly what to ask.

1
Frame The business side defines a specific question.
2
Shape Engineers build a wide table / data model to match.
3
Read The business side reads charts and decides.
BI is like assembling a puzzle when you already know the picture on the box.
A Concrete Question

How Does BI
Answer a Single Question?

A researcher asks: "How has Russian publication output changed recently?"
A typical BI workflow must walk through three steps before giving the first answer.

1
Communicate Requirements

The analyst meets with domain experts and DBAs to agree on sources, time range, "what counts as Russia".

No data has been touched yet — time is already gone.
2
Build the Wide Table

Manually join Publication × Organization by country and year — heavy dependency on schema knowledge.

This step takes the longest and gets reworked the most.
3
Analyze & Visualize

Finally aggregate, slice, draw a trend chart. One question answered.

But if the result raises a new question — the whole pipeline restarts from Step 1.
BI can answer one known question well. But real investigation is rarely just one question — each answer leads to the next.

What BI has to do

The moment the result lands, the next question appears: why is it falling?

Nothing from Step 1 can really be carried forward.
2
Step 2 · Hypothesize a Cause

Next ask: did
funding change too?

What BI has to do

To look at funding, BI needs a brand-new wide table: Publication × Organization × Funding. The schema is static — you can't graft Funding onto Step 1's result. So the analyst loops through the whole "communicate → build → query" cycle again.

Result: Russian institutions had several funding sources in 2020–2021; all of them cut off from 2022. Funding collapse and output decline arrive together.
3
Step 3 · Validate the Hypothesis

Then ask: did EC redirect
to Ukraine?

What BI has to do

Comparing Russia's funding to Ukraine's needs yet another Organization table — the wide table is now a four-way join. Another DBA conversation, another rebuild, another full recompute.

Result: the European Commission fits the pattern — funded both pre-war, cut Russia in 2022, continued Ukraine. Confirmed against the EC's official site.
Three rounds, three wide tables. The Publication × Organization join has been computed 3 separate times — every round pays the same build cost, with zero reuse.

Turkey's mystery, solved — platform isolation is the deeper driver. International databases (OpenAIRE, Crossref, publishers) systematically reduced Russian paper visibility. This is sanction-agnostic, so it hits every country.

Where This Workflow Breaks

Why the Classic Workflow
Can't Carry Exploration

01

Heavy dependence on expert knowledge

Analysts must identify all relevant tables upfront and coordinate with the DBA to build them. Miss or misjudge — the conclusion skews.

02

Wide-table computation is expensive

Materializing multi-table joins on large data is slow and memory-heavy. Step 3's four-way join blows up on OpenAIRE: 193M publications, 51M Pub–Org edges.

03

The schema is frozen

Once the star / snowflake schema is fixed, it can't be modified mid-analysis. New dimension? Start from scratch — exploration can't "think while walking".

04

Intermediate results aren't reusable

In the case study, the Publication–Organization join was computed 3 separate times. Every round starts fresh; the previous table is discarded.

Paradigm Shift

The same case, done differently—
Exploratory BI

Not three wide tables, but one dataset that keeps growing: each step is an increment on the previous result.

The same case, as D1 → D2 → D3: no rebuilds, no duplicated joins.

D1
Step 1: Publication ↔ Organization

Russian publication counts over time.

D2
Step 2: D1 + Funding

Schema evolves in place. No new analysis round — new dimension is grafted onto the previous result.

D3
Step 3: D2 + another Org

Rounds are logically connected. Insights compose across steps, instead of living in three isolated reports.

Two shifts: schema evolves incrementally, and rounds reuse & compose. The rhythm changes from "repeat" to "accumulate".
Data Model & Operators

To Power This Style of Exploration,
We Need a New Foundation

Key observation: in classic BI, "wide table = joining multiple tables" — and table joins are essentially subgraph matching on a graph.
If analysis is fundamentally a graph operation, let's model the data as a HyperGraph directly.
HyperGraph Data Model
  • Vertex: entities — Publication, Organization, Funding…
  • Hyperedge: one pattern-match result — i.e. "a row in the wide table"
  • Grows incrementally: add new dimensions on top; no rebuild
Classic OLAP still works: Drilldown Rollup Slice Dice — the hypergraph can be projected to a relational table at any time, so analysts don't learn anything new.
Three new operators (what they do, not how)
S
Source · Extract from the graph

Given a user-supplied pattern, find all matches in the underlying graph and package them as a hypergraph.

J
Join · Compose two hypergraphs

Merge two hypergraphs by a condition — add a new dimension while keeping what was already there.

V
View · Project back to a table

Flatten the hypergraph into a relational table, handed back to the BI analyst's familiar surface.

Core Contribution — Part 1

Making Exploration Feasible:
Two Performance Walls

Exploratory BI runs Source and Join over and over — and both hit a wall instantly on graph data.

Wall 1 — Source Operator

Subgraph matching is NP-complete

Evidence: finding a 4-path on a graph with only 9,892 vertices takes ~4 minutes under exact enumeration.
For interactive analysis — even 4 seconds feels slow.

Worse: even if matching finishes, materializing every result into a hypergraph blows the memory budget.

Wall 2 — Join Explosion

Hypergraph joins blow up fast

Evidence: on OpenAIRE, a single citation join yields 1.4 billion rows.
Exploration keeps joining — the cascade is unsurvivable.

If the hypergraph isn't pre-sorted, naive join complexity jumps to O(|E1| · |E2|) — effectively unrunnable at scale.

Key insight: BI analysts care about aggregates (COUNT / SUM / trends), not individual records.
If the aggregate can be estimated accurately — materializing the full result set is unnecessary.
Operator 1 · Sampled

Source-Sample

Sidestep Source's NP-complete enumeration — instead of finding all subgraph matches, draw a genuinely uniform random sample.

Trait 1 · No full enumeration

Only r matches are drawn — the combinatorial explosion of subgraph matching simply doesn't happen. Regardless of graph size or density, the cost is one sample's worth.

Trait 2 · The sample is uniform

Every possible match has equal probability of being drawn. This is the foundation for every downstream correctness guarantee — only a uniform sample makes COUNT / SUM unbiased.

Trait 3 · Native property-graph support

We adapt FaSTest's tree sampling to property graphs with vertex / edge attribute filtering — conditions like "Russian institutions", "after 2022" go straight into the pattern query.

Trait 4 · Returns a hypergraph

The output isn't just a count — it's a full hypergraph. Downstream Join / View / Drilldown / Slice plug in seamlessly, as the entry point for further exploration.

Operator 2 · Sampled

Join-Sample

Never materialize the full join result — draw a sample that is statistically equivalent to uniform sampling of the complete join.

Trait 1 · No blow-up materialization

OpenAIRE's 1.4-billion-row join result is never constructed. Join-Sample locates exactly the r rows we need via weighted sampling, skipping the intermediate explosion entirely.

Trait 2 · Equivalent to uniform sampling

Although the mechanics are two-stage weighted, every final join result has equal selection probability. The output is statistically equivalent to uniformly sampling the full join.

Trait 3 · Composes with Source-Sample

Join-Sample's inputs can be Source-Sample outputs, or the output of another Join-Sample. Uniformity survives composition — the basis for reusing results across the whole exploration chain.

Trait 4 · Scales to real graphs

Exact joins routinely OOM or time out on OpenAIRE-scale data; Join-Sample still produces stable answers on the queries where exact algorithms simply can't complete.

Core Contribution — Part 3

End-to-End Unbiasedness

Individual-operator uniformity is easy — the real challenge is: does the sample stay uniform under arbitrary composition?

Theorem (End-to-End)
For any exploration workflow built from
Source-Sample & Join-Sample, —
every hyperedge in the output hypergraph
is a uniform sample of the true result.

Proof sketch: induction on workflow structure. If the last step is Source-Sample — from the lemma directly; if Join-Sample — one more nesting preserves the expectation identity.

Corollary — Unbiased Estimation
∀ hypergraph G built via
Source-Sample / Join-Sample —
COUNT(G): HT estimator is unbiased
SUM(G): HT estimator is unbiased
    (Horvitz-Thompson estimator)

For DISTINCT / QUANTILE / MAX / MIN — a uniform sample is also the best input for GEE, t-digest, etc., with bounded error.

Error Stability Across Multiple Joins

The concern: do repeated joins accumulate error?
Experiments (1–4 joins; path & cycle patterns):

  • COUNT error stays < 1%
  • MAX error stays ~0.01%
  • DISTINCT COUNT in 2–10% — does NOT grow with hops
What This Actually Enables

The core requirement of Exploratory BI — "every step continues on the previous step's sample" — is now provably sound.
Numbers the analyst sees at any point are unbiased estimates of the full-data answer.

Empirical Validation

Speed & Accuracy — Both Deliver

LDBC SNB, three scales (SF1 / SF3 / SF10), default sample size 1M.

End-to-End Speedup 36.7× SF3 / Q2 ·  40s vs. 1475s
COUNT Error < 1% 1M samples ·  all SF1 queries
MAX Error ~0.01% Stable, weakly sensitive to sample size
SF10 Completion 10 / 10 No-sample baseline: 1 / 10 (rest OOM/OOT)
Robustness Beats Raw Speed

At SF10 — the naive baseline fails 9 of 10 queries (OOM or 2-hour timeout); the sampling version completes 10 / 10.

At scale, sampling isn't "optional" — it's required.

Sample-Size Tradeoff

Accuracy gains are steepest from 10K → 100K; 100K → 1M is the knee;
1M → 10M — marginal accuracy, but time grows ~10× linearly.

Default 1M — the sweet spot.
